    üìò Lesson: HTTP Cache-Control ‚Äì Concepts and Browser Testing with FastAPI #

    HTTP caching plays a crucial role in improving performance, reducing latency, and minimizing server load. One of the most versatile headers for managing caching behavior is Cache-Control. In this lesson, we’ll explore how it works from both the server and client perspectives, and demonstrate how to test it using FastAPI and Chrome DevTools.

    üîÑ Why Cache? #

    Caching helps eliminate redundant requests to the server by allowing clients or intermediaries to reuse stored responses. It:

    • Reduces page load times
    • Decreases bandwidth usage
    • Improves scalability and responsiveness

    üîë Understanding Cache-Control #

    ‚úÖ Server-Side Cache-Control #

    • Direction: Response
    • Purpose: Tells clients, browsers, and CDNs how the response should be cached.
    Header Example Description
    Cache-Control: public, max-age=10 Response is cacheable by any cache and remains fresh for 10 seconds.
    Cache-Control: no-store Response must not be cached or stored by any system.

    üì• Client-Side Cache-Control #

    • Direction: Request
    • Purpose: Instructs caches how to treat previously stored responses.
    Directive Description
    no-cache Forces revalidation before using cached data. Cached data may still be stored.
    max-age=0 Immediately treats any cached data as stale and triggers revalidation.
    no-store Prevents caching of the request and its response entirely.
    only-if-cached Uses the cached response only; fails if no cached version exists.

    üîç Note: no-cache allows caching but requires validation, whereas no-store forbids storage altogether.

    üõ†Ô∏è FastAPI Hands-On: Two Endpoints for Caching #

    üìÑ Start Your Server #

    Change directory to 03-http-headers and create a new Python file cache-control.py:

    cd 03-http-headers
touch cache-control.py

    Paste in the following Python code:

    from fastapi import FastAPI, Response
from datetime import datetime

app = FastAPI()

@app.get("/cached")
def cached_resource(response: Response):
    response.headers["Cache-Control"] = "public, max-age=10"
    return {
        "message": "This response is cached for 10 seconds.",
        "timestamp": datetime.utcnow().isoformat()
    }

@app.get("/no-store")
def no_store_resource(response: Response):
    response.headers["Cache-Control"] = "no-store"
    return {
        "message": "This response is never cached.",
        "timestamp": datetime.utcnow().isoformat()
    }

    Use Uvicorn to start your FastAPI app:

    uvicorn cache-control:app --reload

    üß™ Test It Out Using the Browser Console #

    You can test caching behavior directly in the browser using Chrome DevTools:

    1. Open Chrome DevTools (F12)
    2. Go to the Console tab
    3. Use the fetch() API to simulate HTTP requests with custom Cache-Control headers.

    üîÅ Test with max-age=0 #

    Paste below commands in the Console tab:

    fetch("http://localhost:8000/cached").then(res => res.json()).then(console.log);
fetch("http://localhost:8000/cached").then(res => res.json()).then(console.log);
fetch("http://localhost:8000/cached", { headers: { "Cache-Control": "max-age=0" } }).then(res => res.json()).then(console.log);
    • The first two responses should show same timestamp, because the second request is returned by the browser’s cache.
    • Header max-age=0 makes the cache stale, and the request is sent to the server for validation. Because our server code doesn’t do validation, new response instead of cache is returned.

    You should only see two requests logs in the server terminal, and they are from the first and the third requests.

    Browser’s responses:

    {message: 'This response is cached for 10 seconds.', timestamp: '2025-04-13T06:48:13.860974'}
{message: 'This response is cached for 10 seconds.', timestamp: '2025-04-13T06:48:13.860974'}
{message: 'This response is cached for 10 seconds.', timestamp: '2025-04-13T06:48:13.865528'}

    Server terminal responses:

    INFO:     127.0.0.1:57927 - "GET /cached HTTP/1.1" 200 OK
INFO:     127.0.0.1:57927 - "GET /cached HTTP/1.1" 200 OK

    üö´ Test with no-store #

    Paste below commands in the Console tab:

    fetch("http://localhost:8000/no-store").then(res => res.json()).then(console.log);
fetch("http://localhost:8000/no-store").then(res => res.json()).then(console.log);

    Because header no-store is set by the server, the response won’t be cached by the browser. Hence, every request is sent to the server. You should see two different timestamps in the responses and two request logs in the server terminal.

    Browser’s responses:

    {message: 'This response is never cached.', timestamp: '2025-04-13T06:49:49.211935'}
{message: 'This response is never cached.', timestamp: '2025-04-13T06:49:49.216012'}

    Server terminal responses:

    INFO:     127.0.0.1:58336 - "GET /no-store HTTP/1.1" 200 OK
INFO:     127.0.0.1:58336 - "GET /no-store HTTP/1.1" 200 OK
