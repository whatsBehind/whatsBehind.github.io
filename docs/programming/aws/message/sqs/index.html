    Architecture of SQS System #

    Three Main Components #

    • Producers

      Producers in above system are the identities who send message to SQS queue

    • SQS Queue

      SQS Queue is a buffer that stores messages and decouples producers and consumers in the system

    • Consumers

      Consumers are identities in the system poll message from SQS queue

    How to Use SQS Queue #

    1. Producer sends message

      Producer sens message to SQS Queue, and the message will be visible to all consumers

      • Message state - Visible: The message is able to be polled by any consumers. The message is in visible state when
        1. Message is sent to the queue and not pulled by any consumers
        2. Message is not deleted before the visibility timeout runs out
      • Visibility timeout: The time that a message stays in the queue but is hidden to all consumers. Message is supposed to be polled and processed by only one consumer at a time. Hence, once a consumer polls the message, it should not be visible to all consumers

    2. Consumer polls message

      Consumer polls messages (up to 10 in one batch) from the queue and processes the messages. The message will be invisible until 1. Consumer who polls the message completes message processing and deletes the message 2. Or consumer doesn’t delete the message before visibility timeout runs out, and the message becomes visible again

      • Message state - Invisible: The message can’t be polled by any consumers.

    3. Consumer deletes message

      Consumer completes message processing and deletes the message from the queue

    SQS Access Policy #

    Policy that is attached to a SQS queue, defines WHO (the identity) can perform WHAT actions (APIs that are allowed) on this SQS queue

    Cross Account Access Policy # 

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::<Account-B-ID>:root"
      },
      "Action": [
        "SQS:SendMessage",
        "SQS:ReceiveMessage",
        "SQS:DeleteMessage",
        "SQS:GetQueueAttributes",
        "SQS:GetQueueUrl"
      ],
      "Resource": "arn:aws:sqs:<region>:<Account-A-ID>:<queue-name>"
    }
  ]
}
    • The access policy allows Account-B to access SQS queue in Account-A

    SQS Standard Queue #

    Attributes #

    • Unlimited throughput and number of messages in queue
    • Default retention of message: 4 days. Max: 14 days
    • Low latency
    • Limitation of 256 KB per message
    • Can have duplicate messages
    • Could be out of order

    DLQ (Dead Letter Queue) #

    If consumer doesn’t delete the message from the queue before visibility timeout runs out, the message will be visible again and receive count for that message increments.

    We can set up a threshold of how many times a message could go back to the queue. If MaximumReceives threshold is exceeded, the message goes into a dead letter queue

    DLQ is mainly for debugging purpose. It also supports to re-drive failing messages which sends messages from DLQ to SQS queue
