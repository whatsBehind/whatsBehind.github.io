    💾 Docker Mastery Course – Lesson 6 #

    📦 Data Persistence with Volumes and Bind Mounts #

    Welcome to Lesson 6 of the Docker Mastery Course! In this lesson, you’ll learn how to persist data in Docker containers using volumes and bind mounts, and understand the differences between them.

    🎯 Goal #

    By the end of this lesson, you’ll be able to:

    • Understand why container data is ephemeral by default
    • Use volumes to persist data across container lifecycles
    • Use bind mounts to sync data between host and container
    • Explain the differences between volumes and bind mounts clearly
    • Understand how multiple containers can share the same volume
    • Know how Docker handles (or doesn’t handle) concurrent writes

    📘 Concepts #

    1. Why Data Persistence Matters #

    Docker containers are designed to be ephemeral — meaning:

    • If a container is deleted or restarted, any data written inside it is lost.
    • This makes it hard to store logs, databases, or application-generated files.

    To solve this, Docker supports two methods:

    Method Description
    Volumes Docker-managed persistent storage. Recommended for most cases.
    Bind mounts User-specified paths on the host. Great for development workflows.

    2. Volume vs Bind Mount: What’s the Difference? #

    Feature Volume Bind Mount
    Managed by Docker ✅ Yes ❌ No — user defines path
    Host path visible ❌ Abstracted (Docker manages it) ✅ Directly maps to a host folder
    Created by docker volume create or automatically User simply provides a path
    Location /var/lib/docker/volumes/... (Linux/Mac) Anywhere on the host
    Use case Databases, persistent files Syncing source code in development

    🔎 Are bind mounts like symlinks? Not quite — they’re actual kernel-level filesystem mounts. Bind mounts behave like “real-time mirrors” of host folders.

    🔎 Are volumes like managed storage? Yes. Volumes are Docker-controlled storage resources with automatically assigned host paths. You can inspect or back them up, but Docker handles them for you.

    🛠️ Hands-On Practice #

    ✅ Step 1: Create a Volume # 

    docker volume create mydata

    Check:

    docker volume ls
docker volume inspect mydata

    This creates a managed resource with its own storage path.

    ✅ Step 2: Use the Volume in a Container # 

    docker run -it --name voltest -v mydata:/data alpine
    • mydata:/data means: mount Docker volume mydata into /data inside the container
    • Inside, run:
    echo "Hello Volume!" > /data/hello.txt
exit

    Now remove the container:

    docker rm voltest

    Start a new container to access the same volume:

    docker run -it -v mydata:/data alpine cat /data/hello.txt

    ✅ You’ll see the message again — the data persisted.

    ✅ Step 3: Use a Bind Mount #

    Create a folder on your host:

    mkdir ~/docker-test
echo "hello from host" > ~/docker-test/host.txt

    Run this container:

    docker run -it -v ~/docker-test:/data alpine ls /data

    You should see host.txt. This is your local folder mounted into the container at runtime.

    ✅ Key Syntax: What does mydata:/data mean? #

    This is used in the -v option:

    • mydata → volume name (Docker-managed)
    • /data → path inside the container

    So it means: “Mount the Docker volume mydata at /data in the container.”

    For bind mounts, the syntax looks like:

    -v /host/path:/container/path

    Example:

    -v ~/project:/app

    This binds your host’s ~/project folder to the container’s /app directory.

    📍 With volumes, Docker chooses the host path and manages it. With bind mounts, you define the host directory.

    🔄 Volume Sharing and Deletion #

    🧩 Can a Volume Be Used by Multiple Containers? #

    ✅ Yes. A Docker volume can be mounted into multiple containers — they all share the same underlying data.

    Example:

    docker run -d --name c1 -v mydata:/shared alpine sleep 300
docker run -it --name c2 -v mydata:/shared alpine sh

    Both containers see the same files inside /shared.

    You can even mount the same volume at different paths:

    docker run -v mydata:/app1 alpine

docker run -v mydata:/backup alpine

    🚫 What Happens If a Volume Is Deleted? #

    • If a volume is in use, Docker will block deletion:
    docker volume rm mydata
# → Error: volume is in use
    • If not in use, deleting the volume removes its data permanently.
    • Anonymous volumes can be removed automatically when containers are deleted using --volumes.

    ⚠️ Concurrent Writes and Volume Safety #

    Docker does not coordinate concurrent access to volumes.

    • All containers writing to the same volume access the same underlying filesystem.
    • Docker relies on the host OS filesystem to handle concurrent reads/writes.
    Scenario Safe? Notes
    Multiple containers reading ✅ Yes Safe and common
    One writer, multiple readers ✅ Usually Still depends on app-level logic
    Multiple writers ❌ Risky May lead to corruption or race issues

    🧠 If you have multiple writers, use application-level coordination (e.g., file locks, DBs, queues).

    🏁 Checkpoint #

    You now know how to:

    • Create and manage Docker volumes
    • Use volumes to persist data between containers
    • Bind host folders into containers for development
    • Distinguish clearly between volumes and bind mounts
    • Share volumes across multiple containers
    • Handle deletion scenarios and avoid data loss
    • Understand Docker’s lack of built-in concurrency control

    🎉 Your containers are now capable of retaining, sharing, and coordinating data like full-scale applications.
